# 常见售后问题

### 1.内存不足 <a href="#out-of-memory" id="out-of-memory"></a>

**问题概要：**

客户已达到数据库内存限制，并且

1. 如果无法驱逐键，则客户端在将数据写入数据库时​​会收到 OOM（内存不足）错误
2. 正在逐出客户端的键（如果启用了逐出策略），这可能会导致数据库延迟增加

**收集信息：**

1. 从 UI 中的数据库配置页面找出数据库的最大内存限制
2. 找出为数据库设置的逐出策略。
3. 查看当前内存使用情况和使用趋势
4. 从指标/图表中找出驱逐率（如果有）并将其与延迟图表进行比较
5. 查看是否有任何键即将过期

**故障排除步骤：**

询问客户该应用程序是否打算使用此数量，或者它是一个应用程序错误

1. 如果有错误，请客户修复它，如果需要，RedisLabs 支持可以帮助客户
2. 如果没有，则向客户解释问题并将他们重定向到解释驱逐政策的文档，以便他们可以选择适合他们用例的政策。他们还可以简单地将数据库的最大内存限制（直到 50GB 有复制和 25GB 没有复制，他们将需要在数据库上启用集群）增加到所需的水平。客户还可以为密钥设置较低的过期值，以便这些密钥过期足够快并且不会填满数据库。

**潜在的根本原因：**

1. 客户申请问题
2. 最初设置数据库时，客户对最大内存的估计不足

**推荐的解决方案：**

1. 如果客户不打算逐出数据，请要求客户设置适当的逐出策略或增加最大内存限制。
2. 要求客户修复其应用程序中的错误（如果有），或者如果客户在修复错误时需要帮助，请升级到 RedisLabs 支持。

### 2. 响应延迟 <a href="#latency" id="latency"></a>

**问题概要：**

客户应用程序在向 Redis 数据库读取和写入数据时响应时间过长。同时，RedisLabs 仪表板或 Stackdriver 上的图表显示 DB 上的延迟很高，即通常超过 1 毫秒。

**收集信息：**

1. 从 UI（RL 仪表板或 Stackdriver）获取图表以确认数据库有延迟。
2. 从UI采集SLOWLOG或者让客户提供，使用redis命令**SLOWLOG GET 128**
3. 找出元素数量最多的键。[这可以使用redis-cli 中的 --bigkeys 选项](https://redis.io/topics/rediscli#scanning-for-big-keys)来完成 。
4. 找出消耗更多内存的键。这可以使用 redis-cli 中的“--memkeys”选项来完成。
5. 从 UI 中找出数据库的 ops/sec（吞吐量）

**故障排除步骤：**

确认 UI 中的延迟后，首先应该检查的是 SLOWLOG 条目，以查看那里是否存在任何命令。

1. 如果 SLOWLOG 中有命令与延迟增加的时间一致，那么我们需要将这些命令分叉为：
   * 具有更高时间复杂度的 CPU 密集型命令，例如 O(N)。例如 KEYS、SSCAN、SMEMBERS、ZSETS、ZRANGE、ZREVRANGE 等。
   * 时间复杂度较低的非 CPU 密集型命令，如 O(1) 或类似命令。例如 GET、SET、DEL、SCAN、INCR、INFO 等。这些简单命令出现在 SLOWLOG 中的最明显原因是 Redis 进程的高 CPU 使用率导致处理请求的延迟。
2. 如果 SLOWLOG 为空但数据库存在延迟，则很可能是基础架构出现问题。该问题需要上报给 RedisLabs 支持，以便我们进行调查。

如果 UI 中没有延迟，但客户抱怨超时或读取/写入数据库的延迟，那么这就是应用程序/网络延迟，这通常不在 RedisLabs 的控制范围内。它是应用程序和端点之间的延迟。我们 UI 上的延迟图仅显示分片和 DB 端点之间的延迟（请参阅上面的数据平面图）。在这种情况下，要确认应用程序和数据库之间的网络延迟，可以使用 应用程序服务器上的[redis-cli 的 --latency 开关。](https://redis.io/topics/rediscli#monitoring-the-latency-of-redis-instances)

**延迟的根本原因：**

1. 如果在 SLOWLOG 中找到命令，则可能是 redis 使用不当或达到了 DB ops/sec 限制。
2. 如果不是，但 UI 上存在延迟，则很可能是基础设施问题或 redis 中的错误。
3. 如果 UI 中没有延迟，那么这是 Redis 之外的问题，主要是网络或 Redis 客户端配置。

**推荐的解决方案：**

1. 如果在 SLOWLOG 中发现 CPU 密集型命令，那么我们需要指导客户正确使用 Redis，为此应将票证连同初始调查期间收集的所有数据一起转发给 RedisLabs 支持。
2. 如果在 SLOWLOG 中发现非 CPU 密集型命令，则数据库可能已达到数据库当前配置可实现的最大操作数/秒。如需进一步咨询，需要将案例转发给 RedisLabs 支持。
3. 如果以上2个原因都没有找到，那么有可能是infra问题或者redis有bug。在这种情况下，请立即将工单上报给 RedisLabs 支持。
4. 如果存在应用程序延迟，那么客户需要在他们的终端解决它，但谷歌和 RedisLabs 支持应该能够应客户要求提供帮助。客户还可以验证应用服务器是否尽可能靠近数据库（同一可用区）。

### 3.连通性 <a href="#connectivity" id="connectivity"></a>

**问题概要：**

客户应用程序无法连接到 Redis 数据库。

**收集信息：**

1. 来自 UI 的数据库配置。
2. 客户在其应用程序中使用的端点。
3. 客户尝试连接的 IP。

**故障排除步骤：**

1. 检查数据库是否启用了源 IP ACL，如果启用，请检查客户是否从这些 IP/s 连接。
2. 检查数据库是否启用了 SSL，如果启用，客户使用的是正确的客户端证书、CA 证书和客户端证书密钥。
3. 检查客户是否连接到正确的公共端点或专用端点。

**潜在的根本原因：**

1. 客户使用的端点或端口号不正确。
2. SSL 证书不正确。
3. 防火墙或任何网络安全应用程序/设备。
4. 从 SourceIP ACL 中未列出的 IP 进行连接。
5. 潜在的基础设施问题。

**推荐的解决方案：**

1. 确保使用正确的端点和端口。
2. 确保使用正确的 SSL 证书和密钥。
3. 确保应用程序服务器上的端点端口或 IP 未被任何网络安全软件或设备阻止。
4. 有问题的应用程序服务器的 IP 列在 SourceIP ACL 中（如果有）。
